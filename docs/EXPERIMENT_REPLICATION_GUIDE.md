# Experiment Replication Guide

This document explains, in clear step-by-step English, how to replicate the experiment described in this repository.

It covers two execution paths:

1. **Docker path**
   Best option for a local and fast replication of the experiment.
2. **K3s path managed with Rancher UI**
   Best option when the user should manage the deployment visually, without working with `kubectl` from the terminal.

## Scope of the Experiment

The experiment replicates an industrial MLOps lifecycle for a CNC tool-breakage scenario.

The expected behavior is:

1. The platform starts.
2. Historical CNC telemetry is created and stored.
3. A baseline model is trained and registered in MLflow.
4. A production model is deployed to the edge through a signed deployment manifest.
5. The CNC simulator starts sending telemetry.
6. Edge inference produces predictions close to the machine.
7. Drift and health indicators are collected.
8. Retraining, promotion and rollback can be triggered through Airflow and the Enterprise Tier.

## Part A. Replication with Docker

### Step 1. Check prerequisites

You need:

- Docker Desktop installed and running.
- At least 8 GB RAM and 4 CPUs assigned to Docker.
- The following ports available on your machine:
  - `8080` Airflow
  - `5000` MLflow
  - `8888` Jupyter
  - `1880` Node-RED
  - `1883` MQTT
  - `3000` Grafana
  - `8085` Enterprise API
  - `8088` Enterprise UI
  - `9001` MinIO Console
  - `9090` Prometheus
  - `9092` Kafka

### Step 2. Open the project folder

From the repository root, go to the Docker environment:

```bash
cd Docker-Cloud-Fog-Edge
```

### Step 3. Review the local environment file

Open `.env` and check the values.

At minimum, verify:

- `AIRFLOW_UID`
- `AIRFLOW_VERSION`
- `TIMESCALE_VERSION`
- `KAFKA_VERSION`

If you are on macOS, `AIRFLOW_UID=501` is usually correct.

### Step 4. Build and start the full stack

Run:

```bash
docker compose up --build -d
```

This command builds the custom images and starts the complete architecture.

### Step 5. Wait for the initial bootstrap to finish

The first run takes longer than the next ones.

Wait until these actions are complete:

- TimescaleDB is healthy.
- MinIO is available.
- MLflow is available.
- `db-bootstrap` finishes.
- `edge-sync` receives the first production model.
- `cnc-simulator` starts sending telemetry.

You can check the most important services with:

```bash
docker compose ps
docker compose logs db-bootstrap
docker compose logs edge-sync
docker compose logs cnc-simulator
docker compose logs edge-inference
```

### Step 6. Open the main interfaces

After the platform is ready, open these URLs:

- Airflow: [http://localhost:8080](http://localhost:8080)
  - User: `admin`
  - Password: `admin`
- MLflow: [http://localhost:5000](http://localhost:5000)
- MinIO Console: [http://localhost:9001](http://localhost:9001)
  - User: `admin`
  - Password: `password123`
- Grafana: [http://localhost:3000](http://localhost:3000)
  - User: `admin`
  - Password: `admin`
- Enterprise UI: [http://localhost:8088](http://localhost:8088)
- Enterprise API health endpoint: [http://localhost:8085/health](http://localhost:8085/health)
- Node-RED: [http://localhost:1880](http://localhost:1880)

### Step 7. Validate that the baseline experiment was created

Check the following:

1. In **MLflow**, a registered model called `cnc_tool_breakage_classifier` exists.
2. At least one model version is in the `Production` stage.
3. In the **Enterprise UI**, you can see:
   - production model version
   - recent health values
   - recent deployment history
   - recent predictions
4. In **TimescaleDB**, telemetry and inference events are being generated.

### Step 8. Validate the Airflow DAGs

Open Airflow and confirm that these DAGs exist:

- `cnc_bootstrap_reference_architecture_v1`
- `cnc_closed_loop_retraining_v1`
- `cnc_governance_rollback_v1`

If you want to run them manually, trigger them from the Airflow UI.

### Step 9. Run the closed-loop experiment manually

You have two easy options.

**Option A. Use the Enterprise UI**

1. Open [http://localhost:8088](http://localhost:8088)
2. Click `Run Closed Loop`
3. Observe the response message and refresh the page after a few seconds

**Option B. Use the Enterprise API**

Run:

```bash
curl -X POST http://localhost:8085/api/closed-loop/run \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: enterprise-demo-key' \
  -d '{"trigger":"manual-replication"}'
```

### Step 10. Observe the expected outputs

A successful replication should show the following:

- Telemetry is continuously published by the CNC simulator.
- Telemetry is persisted by the Fog Bridge.
- Predictions are generated by the edge inference service.
- Drift reports are stored.
- Model promotion or rollback events appear when triggered.
- Enterprise UI shows updated deployment and prediction data.

### Step 11. Optional manual rollback test

You can test rollback from the Enterprise UI or with the API:

```bash
curl -X POST http://localhost:8085/api/deployments/rollback \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: enterprise-demo-key' \
  -d '{"reason":"manual replication rollback test"}'
```

### Step 12. Stop the environment

When you finish:

```bash
docker compose down
```

If you also want to remove persistent volumes:

```bash
docker compose down -v
```

## Part B. Replication with K3s Managed Through Rancher UI

This section is written for the case where the final user should manage the environment mainly from **Rancher UI**, not from `kubectl`.

### Recommended approach

Use this path if:

- you already have a working **K3s cluster**
- the cluster is registered in **Rancher**
- you want to deploy and monitor the experiment visually through Rancher UI

### Important note before starting

The Rancher workflow avoids `kubectl`, but the custom images still need to exist in a container registry or in the cluster runtime.

Because of that, the only terminal work recommended before opening Rancher UI is:

1. build the images
2. push them to a registry accessible by the cluster, or import them into the cluster nodes

### Step 1. Prepare the required images

From the repository root, build these images:

```bash
docker build -t ind-mlops-airflow:latest -f Docker-Cloud-Fog-Edge/build/airflow/Dockerfile Docker-Cloud-Fog-Edge
docker build -t ind-mlops-mlflow:latest -f Docker-Cloud-Fog-Edge/build/mlflow/Dockerfile Docker-Cloud-Fog-Edge
docker build -t ind-mlops-python-service:latest -f Docker-Cloud-Fog-Edge/build/python-service/Dockerfile Docker-Cloud-Fog-Edge
docker build -t ind-mlops-enterprise-ui:latest -f Docker-Cloud-Fog-Edge/build/enterprise-ui/Dockerfile Docker-Cloud-Fog-Edge
```

### Step 2. Make the images available to the K3s cluster

Choose one of these approaches.

**Approach A. Push to a registry**

This is the recommended option.

Example:

```bash
docker tag ind-mlops-airflow:latest <your-registry>/ind-mlops-airflow:latest
docker tag ind-mlops-mlflow:latest <your-registry>/ind-mlops-mlflow:latest
docker tag ind-mlops-python-service:latest <your-registry>/ind-mlops-python-service:latest
docker tag ind-mlops-enterprise-ui:latest <your-registry>/ind-mlops-enterprise-ui:latest

docker push <your-registry>/ind-mlops-airflow:latest
docker push <your-registry>/ind-mlops-mlflow:latest
docker push <your-registry>/ind-mlops-python-service:latest
docker push <your-registry>/ind-mlops-enterprise-ui:latest
```

If you use this option, update the image names in the K3s manifests before importing them into Rancher.

**Approach B. Import into local K3s runtime**

Use this only if your K3s cluster runs locally and you control the node runtime.

### Step 3. Open Rancher UI

1. Sign in to Rancher.
2. Open the target K3s cluster.
3. Open **Cluster Explorer**.

### Step 4. Import the base manifests first

In Rancher UI:

1. Go to **Cluster Explorer**.
2. Select **Import YAML**.
3. Upload or paste the files from:
   - `K3s-Cloud-Fog-Edge/k8-manifests/00-base/`

Wait until the namespaces and PVCs are created.

### Step 5. Import the Fog Tier manifests

Again in **Import YAML**, upload or paste the files from:

- `K3s-Cloud-Fog-Edge/k8-manifests/01-fog-tier/`

Wait until these workloads are healthy:

- `timescale`
- `minio`
- `fog-bridge`

### Step 6. Import the Cloud Tier manifests

Import the files from:

- `K3s-Cloud-Fog-Edge/k8-manifests/02-cloud-tier/`

This includes:

- Airflow
- MLflow
- the bootstrap job `platform-bootstrap`

Wait until:

- `mlflow` is running
- `airflow` is running
- `platform-bootstrap` is completed successfully

### Step 7. Import the Edge Tier manifests

Import the files from:

- `K3s-Cloud-Fog-Edge/k8-manifests/03-edge-tier/`

Wait until these workloads are running:

- `kafka-broker`
- `mosquitto`
- `nodered`
- `edge-sync`
- `edge-inference`
- `cnc-simulator`

### Step 8. Import the Governance Tier manifests

Import the files from:

- `K3s-Cloud-Fog-Edge/k8-manifests/04-governance/`

Wait until:

- `prometheus` is running
- `grafana` is running

### Step 9. Import the Enterprise Tier manifests

Import the files from:

- `K3s-Cloud-Fog-Edge/k8-manifests/05-enterprise-tier/`

Wait until:

- `enterprise-api` is running
- `enterprise-ui` is running

### Step 10. Verify the bootstrap from Rancher UI

In Rancher UI, inspect:

- **Workloads**
- **Jobs**
- **Pods**
- **Services**

Confirm that:

1. the bootstrap job completed successfully
2. the cloud workloads are healthy
3. the edge workloads are healthy
4. the enterprise workloads are healthy

### Step 11. Open the services from Rancher UI

The exact UI labels can vary slightly by Rancher version, but the workflow is normally:

1. Open **Service Discovery** or **Services**.
2. Select the service you want to inspect.
3. If your environment exposes `LoadBalancer` services, open the external endpoint.
4. If your environment does not expose an external endpoint, use Rancher's **Port Forward** feature.

The main services to open are:

- Airflow
- MLflow
- MinIO Console
- Grafana
- Enterprise UI
- Enterprise API
- Node-RED

### Step 12. Validate the experiment through the UI

A successful replication in Rancher should show:

- the bootstrap job completed
- a production model exists in MLflow
- the edge sync workload is active
- the CNC simulator is publishing telemetry
- the Enterprise UI shows deployment and prediction data
- Airflow shows the versioned DAGs

### Step 13. Trigger the closed-loop experiment

You can do this in two simple ways.

**Option A. Through Enterprise UI**

1. Open the Enterprise UI service.
2. Click `Run Closed Loop`.
3. Refresh after a few seconds.
4. Check whether new deployment or governance events were created.

**Option B. Through Airflow UI**

1. Open the Airflow service.
2. Trigger `cnc_closed_loop_retraining_v1`.
3. Trigger `cnc_governance_rollback_v1` if you want to test rollback governance separately.

### Step 14. Review the results

You should review these outputs:

- **MLflow**
  - registered model versions
  - production stage changes
- **Airflow**
  - DAG execution state
- **Enterprise UI**
  - recent health
  - deployment history
  - recent predictions
- **Grafana / Prometheus**
  - service metrics
  - drift and inference activity
- **Rancher Workloads view**
  - container restarts
  - job completion
  - pod health

### Step 15. Remove the environment from Rancher UI

When the experiment is finished:

1. Open **Cluster Explorer**.
2. Delete the namespaces:
   - `cloud-tier`
   - `fog-tier`
   - `edge-tier`
   - `governance`
   - `enterprise-tier`

This is the simplest visual cleanup path when working with Rancher.

## Troubleshooting

### Docker bootstrap takes too long

Check:

```bash
docker compose logs timescale
docker compose logs mlflow
docker compose logs db-bootstrap
```

### No production model appears in MLflow

Check:

- `db-bootstrap` logs
- MinIO availability
- TimescaleDB availability
- MLflow health

### Enterprise UI opens but shows no data

Check:

- `enterprise-api` is healthy
- the API is reachable
- `db-bootstrap` completed successfully
- telemetry and inference events are present

### Rancher workloads stay in pending state

Check:

- PVC binding status
- whether your custom images are reachable from the cluster
- whether the image names in the manifests match the registry you used

### K3s services do not open from Rancher UI

If no external endpoint is available, use Rancher's **Port Forward** option for the service or the pod.

## Recommended replication order

If you only need one reproducible path for reviewers, use this order:

1. Replicate first with **Docker**.
2. Validate the interfaces and the closed loop locally.
3. Replicate second with **K3s + Rancher UI** for the enterprise/distributed demonstration.

That order is simpler, faster and easier to debug.
